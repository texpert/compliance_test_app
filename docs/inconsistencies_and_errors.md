## Findings details (full descriptions)

[\^obs1]: Verifier API requires valid `App-Id` / `App-Secret`, but obtaining these requires prior registration at `https://priora.saltedge.com/`; this prerequisite was not explicit in initial task docs.

[\^obs2]: Priora portal stopped responding for more than 15 minutes while updating app registration with connector URL and public key.

[\^obs3]: `tpp/register` repeatedly returns HTTP `200` but async email rejects with `given certificate is invalid` for Moldova (`C=MD`) certs — no field-level detail provided across 13 attempts spanning multiple cert profiles (sha1/sha256, CA:TRUE/FALSE, qcStatements variants, ST variants, real PSD2 orgId, near-original BG profiles). **However**, UK attempts (`C=UK`) returned a specific `country UK was not found` error, proving Priora CAN produce field-level rejection messages. This means: (a) `C=MD` passes country validation, (b) something else in the cert profile is rejected, and (c) Priora is not surfacing the specific reason for the Moldova failure despite being capable of doing so. Support emails sent 2026-04-05 and 2026-04-07.

[\^obs4]: Previous docs mixed `berlingroup/v2` base URL text with Berlin Group endpoint paths under `/v1/` (for example `POST /v1/consents`). This is now resolved: for this integration, SCA/AIS Berlin Group calls use `.../berlingroup/v1/...`; `v2` references are for OAuth/Priora session-token APIs and should not be used as the Berlin Group SCA/AIS base path.

[\^obs5]: On 2026-04-06, a clean retry against the newly uploaded public key and cert (`guide_2026-04-06-tpp-register-retry-01`) returned `400 REQUEST_FORMAT_INVALID` with `Header 'Signature' invalid: Malformed signature`. A broader diagnostic sweep on both the new cert and prior `texpert-v2` cert also returned the same `400` for every tested `Signature` variant (DER vs PEM certificate header, hex vs colon fingerprint `keyId`, with/without `tpp-signature-certificate` in signed headers, uppercase `RSA-SHA256`, spacing/order variants, omitted `algorithm`, and `request-target` without parentheses). Diagnostic request IDs: new cert `6b270b52-077c-4713-8dcc-897b497368ad`; prior cert `783397d1-7161-4074-b08a-2edfc849e30b`.

[\^obs6]: The live example curl differs materially from both the header table and common Berlin Group signing practice: it sends a body without the `data` wrapper and without `certificate.value`; uses `Digest: SHA-256=47DE...` (hash of the empty string) despite including a JSON body; shows `TPP-Signature-Certificate` as base64-of-PEM text, not DER; prefixes the header as `Signature: Signature ...`; uses a DN-style `keyId` instead of a fingerprint; signs only `digest date x-request-id`; and the sample line itself contains broken quoting. **Granular isolation on 2026-04-06 revealed the root cause**: changing only the `keyId` from hex fingerprint to DN-style format (e.g., `SN=0,DN=/organizationIdentifier=.../CN=.../...`) shifts the backend from `Header 'Signature' invalid: Malformed signature` to `Header 'Digest' invalid. Expected format: SHA-256={SHA256Base64(requestBody)}` despite the digest being computed from the transmitted body.
